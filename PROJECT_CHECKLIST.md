# imagesPlayEditor 需求与目标清单

## 1. 项目目标
- [x] 基于 Rust + Tauri 实现一个“图片步进指示编辑器”。
- [x] 用户可在单窗口内加载图片并添加标注，按绘制顺序导出步骤图。
- [x] 支持通过动作列表重排步骤、插入额外动作（如清空、保留）、并支持锁定标注防止被清空。

## 2. 核心使用流程
- [x] 打开应用窗口。
- [x] 加载一张图片作为画布底图。
- [x] 添加标注（框选 / 箭头 / 文字）。
- [x] 按拖拽列表调整标注与动作顺序。
- [x] 插入动作：清空之前标注、保留下一标注等。
- [x] 为关键标注添加“锁定”标记，避免被清空动作移除。
- [x] 点击导出后，按动作序列逐步生成多张图片。

## 3. 功能需求

### 3.1 画布与标注
- [ ] 支持图片加载、缩放、平移（基础可用）。
- [x] 框选标注：可配置颜色、线宽、样式。
- [x] 箭头标注：可配置颜色、线宽、长度。
- [x] 文字标注：可配置字号、颜色。
- [x] 标注可编辑、删除、锁定/解锁。
- [x] 标注需记录创建顺序（稳定 ID + 时间序/序号）。

### 3.2 步进动作列表
- [x] 以时间线/列表展示“标注动作序列”。
- [x] 支持拖拽排序。
- [ ] 支持插入系统动作：
- [x] `ClearPrevious`：清空之前标注（锁定标注除外）。
- [x] `KeepNext`：保留下一个标注（用于跳步强调）。
- [ ] 支持动作禁用/启用。

### 3.3 导出
- [x] 按动作序列逐步回放并导出多张图片。
- [x] 导出命名规则可配置（如 `step_001.png`）。
- [ ] 支持导出目录选择。
- [x] 导出结果与预览顺序一致。

## 4. 技术与工程约束
- [x] 技术栈：Rust + Tauri（前后端分离）。
- [x] 前端尽量复用成熟绘图库/交互库（如 Konva/Fabric.js 等，最终选型待定）。
- [x] 单独 Git 仓库（`imagesPlayEditor` 独立初始化）。
- [x] 建立 Lint / Format / 基础质量门禁。
- [x] 代码文件行数约束：单文件不超过 500 行（超出必须拆模块）。
- [x] 模块化导入统一走公共通道（如 `mod.rs` / `index.ts` 统一出口）。
- [ ] 关键逻辑需有最小可行测试（至少导出流程与动作解释器）。

## 5. 分阶段目标

### 阶段 A：项目骨架
- [x] 初始化 Tauri 项目。
- [x] 前端基础 UI（画布区 + 右侧动作列表 + 导出区）。
- [x] 统一模块导出入口规范。

### 阶段 B：标注能力
- [x] 实现框选 / 箭头 / 文字三类标注。
- [x] 实现样式编辑与锁定。
- [x] 建立标注数据模型（可序列化）。

### 阶段 C：动作编排
- [x] 拖拽排序。
- [x] 插入系统动作（清空/保留）。
- [x] 动作解释器（按序回放状态）。

### 阶段 D：导出闭环
- [x] 多步骤图片导出。
- [ ] 命名、路径、错误处理。
- [ ] 回归测试与性能检查。

## 6. 验收标准（首版）
- [x] 用户可从 1 张原图得到 N 张步骤图。
- [x] 任意调整动作顺序后，导出结果稳定可复现。
- [x] 锁定标注不会被清空动作删除。
- [x] 代码结构通过 lint，且无超过 500 行的业务文件。
